# Traffic-Based Route Guidance System (TBRGS)

A deep-learning-powered desktop application that predicts real-time traffic
flow using SCATS sensor data and recommends the **Top 5 optimal routes**
between any two intersections in the **City of Boroondara**, Melbourne.

---

## Features

| Component | Description |
|---|---|
| **Data Processing** | Parses VicRoads SCATS Excel data into clean CSV splits |
| **LSTM** | 2-layer stacked LSTM trained per SCATS site |
| **GRU** | 2-layer stacked GRU trained per SCATS site |
| **Transformer** | Multi-head self-attention encoder trained per SCATS site |
| **Yen's K-Shortest Paths** | Finds top 5 loopless routes using predicted travel times |
| **A\* Search** | Haversine heuristic for initial shortest-path seed |
| **OpenStreetMap GUI** | PyQt5 desktop app with embedded Folium/OSM map |

## How to Run

### Step 1 — Process raw SCATS data  *(skip if CSV files already exist in `data/`)*

Place the raw Excel file at `data/raw/Scats_Data_October_2006.xls`, then:

```bash
python data_processing.py
```

### Step 2 — Train ML models  *(skip if `.pth` files already exist in `models/`)*

```bash
python lstm.py
python gru.py
python transformer.py
```

Each script trains one model per SCATS site (40 sites) and saves weights to
`models/`. Expected training time: ~5–15 minutes per model on CPU.

### Step 3 — Launch the GUI

```bash
python main_gui.py
```

### Using the GUI

1. **Origin SCATS Site** — select a site number (e.g. `2000`)
2. **Destination SCATS Site** — select a different site (e.g. `970`)
3. **Prediction Model** — choose `lstm`, `gru`, or `transformer`
4. Click **"Find Top 5 Routes"** — the system will:
   - Load per-site model weights for every SCATS site
   - Predict next-step traffic flow at each site
   - Convert predicted flow to edge travel times (parabolic formula)
   - Run Yen's K=5 to find the 5 fastest routes
   - Display routes and travel times in the left panel
   - Render all 5 routes colour-coded on the OpenStreetMap

> **Fallback:** If a site's model weights are not found, the system uses
> 300 veh/15 min (free-flow) so routing always remains functional.

## Travel Time Formula

From *Traffic Flow to Travel Time Conversion v1.0*:

```
flow = -1.4648375 × speed² + 93.75 × speed

flow ≤ 351 veh/hr  →  speed = 60 km/h  (speed limit cap)
flow > 351 veh/hr  →  under-capacity (green) root:
                       speed = (-B - √(B² + 4·A·flow)) / (2·A)

travel_time (min) = (distance_km / speed_km_h) × 60 + 0.5
```

The **0.5-minute** constant is the average intersection delay (30 s).  
ML predictions are in **veh/15 min** → multiplied by 4 to get **veh/hr**.

## Algorithm Reference

### Yen's K-Shortest Loopless Paths

1. Find the shortest path with **A\*** (Haversine heuristic) → **A[0]**
2. For k = 1 … K−1:
   - For each spur node along **A[k−1]**:
     - Remove edges used by confirmed paths with the same root prefix
     - Remove root-prefix nodes to prevent loops
     - Run Dijkstra spur → destination on the pruned graph
     - Add `root_path + spur_path` to a candidate min-heap
   - Pop the cheapest candidate → **A[k]**
3. Return K paths sorted ascending by total travel time
